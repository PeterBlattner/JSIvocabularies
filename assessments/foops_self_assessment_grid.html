Findable — can users and machines discover this vocabulary? 0 / 7
The vocabulary has a resolvable URI — typing it into a browser returns a readable page
Basic HTTP resolution test
FIND1
The vocabulary URI is persistent — managed via w3id.org, purl.org, or a documented long-term policy
Not just a server URL that could break on infrastructure change
FIND2
A version IRI exists and resolves — users can access a specific release, not just "latest"
e.g. https://w3id.org/myvocab/1.2/
FIND3
The vocabulary's own identifier (owl:ontologyIRI or skos:ConceptScheme URI) matches its published URI
No mismatch between declared ID and access location
FIND4
Minimum metadata is present: title, description, version number, creation date
Without these, automated catalogues cannot index the vocabulary
FIND5
The vocabulary's namespace prefix is registered in a public prefix registry (prefix.cc or Linked Open Vocabularies)
Allows short-form references (e.g. vim: instead of full URI)
FIND6
The vocabulary is registered in at least one public catalogue or repository (LOV, a domain registry, or a data portal)
Enables discovery by systems that don't already know the URI

The vocabulary URI uses HTTP or HTTPS — open, standardised protocol

ACC1

Single URI, multiple representations
ACC2

Accessible to people, not just machines
ACC3

Long-term availability commitment
ACC4

I
Interoperable — can it be combined with other vocabularies by humans and machines? 0 / 6
The vocabulary is available in at least one RDF serialisation (Turtle, JSON-LD, or RDF/XML)
Minimum interoperability threshold
INT1
Vocabulary metadata uses widely adopted vocabularies: Dublin Core (dcterms:), Schema.org, PROV-O, or DCAT
Not a proprietary metadata schema
INT2
Terms are structured using W3C SKOS (skos:prefLabel, skos:definition, skos:broader/narrower) or OWL
Standard concept model — not custom properties
INT3
The vocabulary explicitly extends, imports, or maps to at least one external vocabulary
e.g. skos:exactMatch, owl:imports, rdfs:subClassOf an external class
INT4
Preferred labels (skos:prefLabel) are provided in at least one language with an explicit language tag
e.g. "unit of measurement"@en not just a plain literal
INT5
Multiple serialisation formats are provided (e.g. both Turtle and JSON-LD), or a SPARQL endpoint is available
Different downstream systems prefer different formats

Reusable — are terms described, licensed, and documented for reuse without contacting the authors? 0 / 6
A license is explicitly stated and resolvable as a URI (e.g. https://creativecommons.org/licenses/by/4.0/)
Machine-readable license — not just "free to use" in plain text
REUSE1
Extensive metadata is provided: creator (linked to ORCID/ROR), contributor, publisher, creation and modification dates
Not just a title and version — full provenance chain
REUSE2
Every term has a human-readable label (skos:prefLabel or rdfs:label) and a definition (skos:definition or rdfs:comment)
No unlabelled or undefined terms
REUSE3
Provenance metadata is present: source of terms (dcterms:source), version history, and change notes (skos:changeNote)
Users can trace where definitions came from
REUSE4
An HTML documentation page is generated and published (e.g. via WIDOCO or equivalent) for human browsing
Separate from the machine-readable RDF file
REUSE5
Semantic versioning is applied (Major.Minor.Patch) and the current version is declared in the vocabulary metadata
Users can cite a specific version unambiguously
